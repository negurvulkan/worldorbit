First steps: what a WorldOrbit document looks like

A WorldOrbit document is plain text. You usually start with a schema line, define a system, and then add objects. Objects can be simple at first and become richer later with orbital data, labels, notes, and presentation defaults.

1

Start with the schema

Use schema 2.6 for new documents. schema 2.5 and earlier still load, but 2.6 adds support for galactic-scale units (ly, pc, kpc), new masses (mj), and specialized phenomenon kinds like galaxies and black holes.

2

Define one system

A file usually contains one system. Think of it as the root container for everything else.

3

Add objects and placement

Then add stars, planets, moons, structures, or belts and describe how they are positioned.

Examples from simple to advanced

These examples are deliberately staged. Start with the smallest possible diagram, then move on to a richer system, and finally look at a more complete representation of our solar system.

One system and one star. This is the simplest valid starting point.

Example 2 — A small star system with planets and a moon

This adds orbit data, a moon, a small station at a Lagrange point, and a named event snapshot.

Sourceintermediate
schema 2.6

system Iyath
  title "Iyath System"

defaults
  view orthographic
  scale presentation
  preset atlas-card
  theme atlas

viewpoint inner
  projection perspective
  camera
    azimuth 28
    elevation 20
    distance 5

group inner-system
  label "Inner System"

object star Iyath

object planet Naar
  orbit Iyath
  semiMajor 1.18au
  eccentricity 0.08
  angle 28deg
  inclination 24deg
  phase 42deg
  atmosphere nitrogen-oxygen
  groups inner-system

object moon Leth
  orbit Naar
  distance 384000km
  phase 140deg
  groups inner-system

object structure Relay-One
  at Naar:L4
  kind relay
  groups inner-system

event eclipse-window
  kind solar-eclipse
  target Naar
  participants Iyath Naar Leth
Rendered result: a system with a star, one planet, a moon, a relay at a special orbital position, and a named event entry.

Example 3 — A more complete solar system

This example combines most of the current Schema 2.5 atlas surface in one place: comments, groups, filtered viewpoints, orthographic and perspective camera setups, a relation, validation hints, typed lore blocks, render hints, and a curated eclipse event with explicit event poses. It is intentionally rich so you can see how WorldOrbit scales from tiny diagrams to denser orbital atlases.

Beginner tips

These habits make WorldOrbit easier to learn and easier to maintain.

Start small

Define the system and one star first. Then add one planet at a time. It is easier to debug small steps than one huge document.

Use defaults later

Presentation settings like defaults, viewpoint, and annotation are useful, but you do not need them for your first diagram.

Think in layers

There is the system itself, the objects inside it, and optional presentation details. Keeping these layers in mind makes the syntax easier to understand.

What beginners usually need first
  • schema 2.6
  • system Name
  • object star Name
  • object planet Name with orbit and semiMajor
  • Optional: a moon, a belt, a structure, or a named event

FAQ

Short answers to the most common beginner questions.

Do I always need schema 2.6?

For new documents, yes. Older schema 2.5 and schema 2.1 atlases still work, but 2.6 is the recommended explicit header for galactic-scale features.

Do I have to use every parameter?

No. WorldOrbit is flexible. Many diagrams start with just a system, a star, and one or two planets.

What is the difference between orbit and at?

orbit places an object on a normal orbit. at places it at a fixed reference position, such as a Lagrange point.

How do I describe eclipses or other special moments?

Use a Schema 2.5 event section. Add target and participants, then optionally add a positions block with one or more pose <objectId> entries when you want a curated viewer or Studio snapshot.

Can I use WorldOrbit in Markdown?

Yes. That is one of its core strengths. It is designed for Markdown-native worldbuilding and documentation workflows.
